## 🔹 1. Контекст и постановка задачи

### 📋 Текущая ситуация
На главной странице приложения все пользователи видят одинаковый баннер с топ-10 популярных фильмов за день.

**Проблемы текущего подхода:**
- Новые пользователи могут не найти релевантный контент среди общепопулярных фильмов
- Постоянные пользователи видят повторяющийся контент → снижается вероятность повторного просмотра
- Упускается возможность увеличить время просмотра через индивидуальные рекомендации

### 🎯 Цель проекта
Оценить, приводит ли персонализация баннера (на основе истории просмотров и жанровых предпочтений) к статистически значимому изменению ключевых метрик, и принять обоснованное решение о внедрении.

## 🔹 2. Гипотеза теста

### Формулировка (двусторонняя)

> **Если** мы заменим универсальный баннер на персонализированный (формируемый на основе истории просмотров и жанровых предпочтений пользователя),  
> **то** ключевая метрика вовлечённости (CTR баннера) **изменится** (может вырасти или снизиться),  
> **потому что**:  
> - 📈 *Потенциальный рост*: релевантный контент повышает вероятность клика и просмотра
> - 📉 *Потенциальный спад*: алгоритм может ошибаться, показывать «странное» или сужать выбор, вызывая негатив

### Формальная запись
```
H₀: CTR_B = CTR_A  (эффекта нет)
H₁: CTR_B ≠ CTR_A  (есть изменение в любую сторону)
```

> ✅ **Почему двусторонняя?** При инвестиции в алгоритмические рекомендации критично детектировать и негативный эффект (например, «пузырь фильтров» или снижение доверия).

## 🔹 3. Метрики теста

### ✅ Метрика успеха (Primary Success Metric)

| Метрика | Формула | Почему выбрана |
|---------|---------|---------------|
| **CTR баннера** | `Клики по баннеру / Показы баннера × 100%` | Прямая реакция пользователя на рекомендацию; быстро измеряется; чувствительна к изменениям |

### 🛡️ Контрольные метрики (Guardrails)

| Метрика | Зачем отслеживать | Критерий «тревоги» |
|---------|------------------|-------------------|
| **Среднее время просмотра после клика** | Не стали ли пользователи закрывать фильм через 2 минуты? | Снижение >5% |
| **Конверсия клик → просмотр** | Не растёт ли количество «пустых» кликов? | Снижение >3% |
| **Отток пользователей (Churn D7)** | Не начали ли пользователи реже возвращаться? | Рост >2% |
| **Выручка на показ** | Не упала ли монетизация? | Снижение >5% |

### 🔍 Информативные метрики (Diagnostic)

| Метрика | Зачем |
|---------|-------|
| **CTR по устройствам** (mobile / desktop / TV) | Работает ли персонализация одинаково хорошо на всех платформах? |
| **CTR по типам пользователей** (новые / постоянные) | Не ухудшился ли опыт для новых пользователей («холодный старт»)? |
| **Разнообразие жанров в кликах** | Не сузился ли «кругозор» пользователя из-за алгоритма? |
| **Динамика CTR по дням** | Стабилен ли эффект или это «эффект новизны»? |

## 🔹 4. Дизайн эксперимента

### 🧪 Группы и рандомизация

| Группа | Логика баннера | Доля трафика |
|--------|---------------|--------------|
| **A (контроль)** | Топ-10 популярных фильмов за день (универсальный) | 50% |
| **B (тест)** | Персонализированный список на основе:<br>• История просмотров пользователя<br>• Предпочтения по жанрам<br>• Поведение похожих пользователей (collaborative filtering) | 50% |

**Единица рандомизации:** `user_id`  
→ Пользователь видит одну и ту же версию баннера на протяжении всего теста (консистентность).

**Стратификация:** Убедиться, что в обеих группах пропорционально представлены:
- Новые пользователи (регистрация < 7 дней)
- Постоянные пользователи
- Разные устройства (mobile / desktop / smart_tv)
- Разные регионы

### 📅 Длительность теста

| Параметр | Обоснование |
|----------|-------------|
| **Минимум** | 14 дней (захватить полный недельный цикл: будни + выходные) |
| **Рекомендуется** | 21 день (учесть эффект новизны и накопить данные по повторным просмотрам) |
| **Критерий остановки** | Достигнута плановая выборка + статистическая значимость + стабильность эффекта в динамике |

## 🔹 5. Расчёт размера выборки

### 📋 Параметры расчёта

| Параметр | Значение | Пояснение |
|----------|----------|-----------|
| Базовый CTR (p₁) | 8% | По историческим данным |
| Относительный MDE | 10% | Минимальный детектируемый эффект |
| Целевой CTR (p₂) | 8.8% | 8% × 1.10 |
| Абсолютный прирост | 0.8 п.п. | 8.8% − 8% |
| Уровень значимости (α) | 0.05 | Стандарт для индустрии |
| Мощность теста (1−β) | 0.80 | Рекомендуемое значение |
| Тип теста | Двусторонний | Консервативный подход |

### 🧮 Код расчёта в Python (statsmodels)

```python
from statsmodels.stats.power import zt_ind_solve_power
from statsmodels.stats.proportion import proportion_effectsize
import numpy as np

# Параметры
baseline_ctr = 0.08
relative_mde = 0.10
target_ctr = baseline_ctr * (1 + relative_mde)  # 0.088

# Размер эффекта Коэна
effect_size = proportion_effectsize(baseline_ctr, target_ctr)  # ~0.02885

# Расчёт выборки
n_per_group = zt_ind_solve_power(
    effect_size=effect_size,
    alpha=0.05,
    power=0.80,
    ratio=1.0,
    alternative='two-sided'
)

print(f"✅ Выборка на группу: {int(np.ceil(n_per_group)):,}")
# Вывод: 18,863 пользователя
```

### 📊 Результаты расчёта

| Показатель | Значение |
|------------|----------|
| Размер эффекта (Коэна h) | 0.02885 |
| **Выборка на группу** | **18 863 пользователя** |
| **Общая выборка (A + B)** | **37 726 пользователей** |
| При 10 000 показов/день | ~3.8 дней на набор |
| **Рекомендуемая длительность** | **14 дней** (с учётом недельного цикла) |

> 💡 **Фактические данные проекта:** ~100 000 пользователей на группу → избыточная мощность, что позволяет провести устойчивый сегментный анализ.

## 🔹 7. Критерии принятия решения

| Исход теста | Статистический порог | Бизнес-решение |
|-------------|---------------------|----------------|
| **✅ Успех** | CTR вырос ≥10%, p < 0.05, guardrails стабильны | Внедрить персонализацию на 100% трафика; запланировать итерации по улучшению алгоритма |
| **⚠️ Неоднозначно** | Рост <10% ИЛИ статистическая незначимость, но тренд позитивный | Продлить тест на 7 дней ИЛИ запустить на 20% трафика для сбора дополнительных данных |
| **❌ Провал** | CTR не изменился или снизился, p < 0.05 | Не внедрять; проанализировать причины (качество алгоритма? данные? сегменты?) |

## 🔹 8. Потенциальные риски и митигация

| Риск | Последствия | Митигация |
|------|-------------|-----------|
| **Эффект новизны** | Пользователи чаще кликают на «новинку», но эффект временный | Сравнить результаты 1-й и 2-й недели; отложить решение до стабилизации |
| **Холодный старт** | Новым пользователям нечего персонализировать → хуже, чем топ-10 | Для новых пользователей показывать гибридную версию (топ + жанровые предпочтения региона) |
| **«Пузырь фильтров»** | Алгоритм показывает только один жанр → пользователь скучает | Добавить метрику «разнообразие рекомендаций»; настроить баланс релевантности и новизны |
| **Технические сбои** | Алгоритм не сработал → пустой баннер или ошибка | Предварительное тестирование на 1% трафика; мониторинг % ошибок в реальном времени |
| **Каннибализация** | Пользователи смотрят рекомендованное, но меньше — другой контент | Отслеживать общее время просмотра за сессию, а не только клики по баннеру |

## 🔹 9. Рекомендации для стейкхолдеров

### 💼 Для бизнеса (руководство, финансы)
- Персонализация — стратегическое направление: даже +5% к CTR может дать миллионы дополнительных просмотров в год.
- Внедрять только после подтверждения устойчивого эффекта и отсутствия негатива по контр-метрикам.
- Подготовить бюджет на поддержку и развитие алгоритма (инфраструктура, ML-инженеры).

### 🛠️ Для продукта (продукт-менеджер, разработка)
- Если тест успешен: запустить итерации — тестировать разные алгоритмы (content-based vs collaborative filtering), добавлять объяснения («Потому что вы смотрели…»).
- Если тест неудачен: провести качественный анализ (опросы, сессии) — понять, что именно не понравилось пользователям.
- Реализовать механизм A/B-тестирования для будущих улучшений рекомендаций.

### 📣 Для маркетинга (трафик, коммуникации)
- Не менять рекламные креативы до конца теста — это «загрязнит» выборку.
- Если функция внедряется: подготовить коммуникацию, объяснить пользователям ценность персонализации.
- Протестировать гипотезу на других точках контакта: email-рассылки, пуш-уведомления.
